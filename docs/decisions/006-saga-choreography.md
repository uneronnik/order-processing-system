# 006 — Saga-хореография вместо оркестрации

## Контекст
Заказ проходит через несколько сервисов (order → inventory → payment → notification). Нужен механизм координации распределённой транзакции.

## Решение
Saga-паттерн в варианте хореографии — сервисы общаются через Kafka-события, без центрального координатора.

## Обоснование
- Нет единой точки отказа — каждый сервис самостоятельно реагирует на события
- Сервисы слабо связаны — знают только о событиях, не о друг друге
- Проще в реализации при небольшом количестве шагов (3–4 шага в саге)
- Kafka обеспечивает надёжную доставку и возможность перечитать события

## Компенсации
- Нет товара → Order переходит в CANCELLED
- Оплата не прошла → Inventory отменяет резерв → Order переходит в CANCELLED

## Альтернативы
- Оркестрация (отдельный Saga-координатор) — лучше при 5+ шагах или сложных ветвлениях, но добавляет ещё один сервис и единую точку отказа
- Распределённые транзакции (2PC) — не подходят для микросервисов из-за жёсткой связности и проблем с производительностью
